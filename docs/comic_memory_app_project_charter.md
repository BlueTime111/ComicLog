# 漫画剧情记忆与分享 App｜产品立项文档

## 1. 项目概述
本项目拟建设一个开源的漫画剧情记忆与分享工具。它不提供漫画资源本身，而是围绕“作品—章节—总结—回顾—分享”建立一层独立的数据与产品能力，帮助追更用户在继续阅读前，快速回想此前剧情、角色关系与关键转折。

项目的核心价值不在“看漫画”，而在“重新进入漫画”。当用户间隔数天、数周甚至数月后再次打开某部作品时，可以在极短时间内想起自己已经读到哪里、上一话讲了什么、最近几话发生了哪些关键事件。

**立项结论：** 本产品应优先验证“章节级总结 + 总结包分享/导入”的闭环，而不是把资源投入在 OCR 或复杂 AI 自动化上。只要分享与增量维护机制成立，产品就具备可持续的数据积累和社区协作价值。

## 2. 背景与问题定义
- 真实痛点：长期追更会导致用户遗忘剧情细节、角色动机、伏笔位置和已读进度，重新进入阅读状态的门槛很高。
- 现有替代方案不足：用户只能依赖群聊、百科、搜索引擎、论坛讨论或自己零散做笔记，信息分散、剧透严重且不以“继续阅读前回顾”为中心。
- 个人维护成本过高：若要求每个用户从第 1 话开始独立维护整部作品的章节总结，长期坚持的概率很低。
- 因此需要共享机制：通过导出、分享、导入“总结包”，让一个人的维护成果成为其他人的起点，从“个人笔记”升级为“社区协作的记忆层”。

## 3. 产品定位
- **产品一句话**：一个开源的漫画剧情记忆与分享工具，帮助用户记录、回顾并交换某部漫画的章节级总结。
- **目标对象**：长期追更用户、补番/补漫用户、喜欢整理剧情的重度读者、二创/考据型用户。
- **不做什么**：不分发漫画资源，不提供盗版内容，不以全文 OCR 转录为公开内容主体。
- **核心资产**：可共享、可版本化、可持续维护的章节总结数据，以及围绕它形成的分享机制。

## 4. 用户画像与使用场景
### 4.1 用户画像
- 追更党：同时追多部作品，更新间隔长；诉求是在开读前 10 秒内想起上一话与篇章主线。
- 补漫党：一次性补很多话，人物与设定容易混淆；诉求是按篇章 / 角色快速建立记忆框架。
- 整理型用户：愿意做总结并分享给别人；诉求是高效维护、导出、更新、被他人导入。
- 轻量用户：不想手写长文；诉求是一句话总结、直接导入他人的总结包。

### 4.2 典型使用场景
1. 追更前回顾：用户打开某部漫画，先看到上一话一句话总结、最近 3 话回顾、当前篇章主线，再决定继续阅读。
2. 导入现成总结：用户发现自己想补某部长篇作品，但不想从零整理，直接导入别人维护到最新话的总结包。
3. 在共享包基础上二次维护：用户导入一份公共总结后，只对自己特别在意的话数补充私人备注，不影响公共内容。
4. 朋友之间私下分享：用户维护了某部作品的优质总结，可导出文件或生成链接分享给朋友导入。

## 5. 产品目标
- P0：验证用户是否愿意使用章节级总结进行回顾，验证分享/导入是否显著降低使用门槛。
- P1：建立共享层与私人层分离的数据架构，支持总结包的版本化与后续更新。
- P2：形成开源项目基础能力，便于社区贡献、部署和二次开发。
- P3：在验证产品闭环后，再引入 OCR / AI 自动摘要作为提效层。

## 6. 核心产品方案
- 章节级总结：每话支持简短摘要、关键词、主要角色、标签与已读状态。
- 读前回顾：在用户继续阅读前，提供上一话摘要、最近几话串联回顾和当前篇章摘要。
- 总结包分享：以结构化数据包为单位导出某部漫画的总结，而不是导出一段散装文本。
- 导入与冲突处理：导入时提示覆盖范围、新增话数、更新话数和冲突话数，并允许仅导入缺失章节或覆盖共享层。
- 共享层 / 私人层：公共总结与用户个人补充分层存储，后续更新时能同步公共层而不覆盖私人备注。
- 搜索：支持按角色、事件、关键词搜索已维护内容，降低二次回忆成本。

**设计原则：** 任何一话的记录动作都应尽量压缩在 15 秒以内；产品优先追求“重新进入剧情足够快”，而不是“整理得足够完整”。

## 7. MVP 范围定义
### 进入 MVP
- 作品库与章节记录
- 最近几话回顾
- 总结包导出 / 导入
- 共享层 / 私人层合并

### 后续版本
- 多端账号与云同步
- OCR / AI 自动摘要
- 公共广场 / 排行榜

## 8. 关键产品边界
- 版权边界：产品不托管漫画原图，不以对话全文转录为公共内容主体，公开分享以用户确认后的摘要和结构化标签为主。
- 剧透边界：所有回顾内容都必须绑定用户已读范围，不应默认展示未读章节信息。
- 开源边界：开放代码与部署方案，数据层面允许用户导出自己的总结包，但不把社区内容强绑定到单一商业平台。

## 9. 技术路线建议
- 客户端：React Native + Expo + TypeScript；优先移动端，后续可扩展 Web 管理页。
- 本地优先：使用 SQLite 作为本地数据底座，确保离线可用与无账号可用。
- 后端（后续）：使用 Supabase 承担鉴权、数据库、对象存储和分享链接能力。
- 数据结构：以作品、章节、总结包、包版本、用户私人备注为核心实体。
- AI 能力：MVP 不依赖 AI；后续可在移动端或服务端引入 OCR 与摘要草稿生成。

## 10. 风险与应对
- 用户录入成本高：默认一句话总结 + 标签模板，先用导入降低冷启动。
- 共享内容质量参差不齐：显示作者、覆盖范围、更新时间，后续引入收藏/订阅/评分。
- 共享与私人笔记混杂：从第一版开始就做双层数据模型。
- 版权理解不清：公开分享仅限摘要与结构化数据，不托管原图。
- 过早投入 OCR：在分享闭环验证前，将 AI 定位为后续增强项。

## 11. 阶段规划与里程碑
- Phase 0：方案收敛；完成立项、PRD、数据模型与原型草图。
- Phase 1：MVP 开发；完成作品库、章节总结、回顾、导入导出、共享/私人层。
- Phase 2：分享增强；完成链接分享、订阅更新、包版本管理。
- Phase 3：AI 提效；完成截图 OCR、摘要草稿、角色/事件抽取。

## 12. 成功指标
- 首周导入率
- 首周回顾使用率
- 总结包分享率
- 人均维护深度
- 共享包复用率

## 13. 立项建议
建议按“总结包协议优先、导入闭环优先、AI 后置”的原则推进 MVP。只要产品能在不提供漫画资源的前提下，让用户快速回顾剧情并复用他人的整理成果，就已经具备明确价值。

立项后应立即进入两项工作：
1. 完成 PRD 和数据模型的冻结版本。
2. 形成低保真原型并用 3–5 个典型用户场景做快速走查。
